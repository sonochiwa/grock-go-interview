# Основы обработки ошибок

## Обзор

```go
type error interface {
    Error() string
}
```

Ошибка в Go — просто интерфейс с одним методом. Нет try/catch, нет исключений. Ошибки — обычные значения.

## Концепции

### Идиоматичная обработка

```go
result, err := doSomething()
if err != nil {
    return fmt.Errorf("doSomething failed: %w", err)
}
// использовать result

// Создание простых ошибок
err1 := errors.New("something went wrong")
err2 := fmt.Errorf("user %d not found", userID)
```

### Почему Go выбрал явные ошибки

1. **Видимость** — каждая точка отказа видна в коде
2. **Контроль** — разработчик решает, как обработать каждую ошибку
3. **Нет скрытых путей выполнения** — в отличие от exceptions, которые "пролетают" через стек
4. **Производительность** — нет overhead на раскрутку стека

### Правило: handle or return

```go
// ХОРОШО: обрабатываем
if err != nil {
    log.Printf("warning: %v", err)
    // продолжаем с fallback
}

// ХОРОШО: пробрасываем с контекстом
if err != nil {
    return fmt.Errorf("loading config: %w", err)
}

// ПЛОХО: игнорируем
result, _ := doSomething() // куда делась ошибка?

// ПЛОХО: логируем И возвращаем (двойная обработка)
if err != nil {
    log.Printf("error: %v", err) // залогировали
    return err                     // И вернули — вызывающий код тоже залогирует!
}
```

## Частые вопросы на собеседованиях

**Q: Почему Go не использует exceptions?**
A: Явные ошибки заставляют обрабатывать каждый случай. Exceptions могут быть "пойманы" далеко от места возникновения, усложняя понимание потока управления.

**Q: Когда можно игнорировать ошибку?**
A: Практически никогда в продакшн-коде. Допустимо для fmt.Fprintf в логгере, Close() после успешного чтения (но лучше проверить).

## Подводные камни

1. **`_` для ошибки** — тихая бомба. Используй `//nolint:errcheck` если осознанно игнорируешь.
2. **Проверяй ошибку ДО использования результата** — результат может быть невалиден при err != nil.
