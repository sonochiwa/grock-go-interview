# Go Scheduler (GMP)

## Обзор

Go scheduler мультиплексирует горутины (G) на OS потоки (M) через логические процессоры (P). Это M:N планирование — M горутин на N потоков ОС.

## GMP модель

```
        G G G         G G G         G G G
         \|/           \|/           \|/
    ┌─────────┐   ┌─────────┐   ┌─────────┐
    │  P (LRQ)│   │  P (LRQ)│   │  P (LRQ)│   P = Processor (GOMAXPROCS)
    └────┬────┘   └────┬────┘   └────┬────┘   LRQ = Local Run Queue
         │             │             │
    ┌────┴────┐   ┌────┴────┐   ┌────┴────┐
    │    M    │   │    M    │   │    M    │   M = OS Thread (Machine)
    └─────────┘   └─────────┘   └─────────┘

              G G G G G G G
              ───────────────
              GRQ (Global Run Queue)
```

### G (Goroutine)

```go
// runtime/runtime2.go (упрощённо)
type g struct {
    stack       stack   // стек горутины
    stackguard0 uintptr // для проверки переполнения стека
    m           *m      // текущий M (или nil)
    sched       gobuf   // контекст планирования (SP, PC, etc.)
    atomicstatus atomic.Uint32 // состояние
    goid         uint64  // ID горутины
}

// Состояния горутины:
// _Gidle      — только что создана
// _Grunnable  — в очереди, готова к выполнению
// _Grunning   — выполняется на M
// _Gsyscall   — в системном вызове
// _Gwaiting   — заблокирована (канал, мьютекс, etc.)
// _Gdead      — завершена
```

### M (Machine / OS Thread)

```go
type m struct {
    g0      *g     // системный стек для scheduling
    curg    *g     // текущая горутина
    p       *p     // текущий P (может быть nil)
    spinning bool  // ищет работу
}
```

### P (Processor)

```go
type p struct {
    m           *m          // текущий M (может быть nil)
    runqhead    uint32      // начало локальной очереди
    runqtail    uint32      // конец локальной очереди
    runq        [256]guintptr // локальная очередь (ring buffer, 256 горутин)
    runnext     guintptr    // следующая горутина (fast path)
    mcache      *mcache     // кеш аллокатора для этого P
}
```

## Как работает планирование

### Запуск горутины

1. `go func()` → создать G, поместить в `runnext` текущего P (или в LRQ)
2. Если LRQ полна (256) → половину перенести в GRQ

### Scheduling loop (schedule())

```
findRunnable():
  1. Проверить runnext P → есть → выполнить
  2. Проверить LRQ P → есть → выполнить
  3. Проверить GRQ → есть → взять
  4. Проверить netpoll (I/O) → есть → выполнить
  5. Work stealing: украсть половину LRQ у случайного P
  6. Ничего не найдено → М становится idle
```

### Work Stealing

```
P1 (пуст) крадёт у P2:
  1. Выбрать случайный P
  2. Заблокировать его LRQ
  3. Забрать половину горутин
  4. Разблокировать
```

Это обеспечивает балансировку нагрузки без центрального планировщика.

### Системные вызовы

```
Горутина вызывает syscall:
  1. G переходит в _Gsyscall
  2. P ОТСОЕДИНЯЕТСЯ от M (P может найти другой M или создать новый)
  3. M блокируется в syscall
  4. Когда syscall завершается:
     - M пытается найти свободный P
     - Если нет P → G помещается в GRQ, M становится idle
```

### Вытесняющее планирование (Go 1.14+)

```
До 1.14: кооперативное
  - Горутина уступает при: вызове функции (stack check), channel op, syscall
  - Tight loop без вызовов → блокирует P

С 1.14: асинхронное вытеснение
  - sysmon (системный монитор) обнаруживает горутину, работающую > 10ms
  - Отправляет SIGURG (Unix) потоку M
  - Signal handler устанавливает preemption flag
  - Горутина прерывается при следующем safe point
```

### sysmon — системный монитор

```go
// Отдельный M без P, работает в фоне:
// - Проверяет горутины в syscall > 10ms → отнимает P
// - Проверяет горутины running > 10ms → preemption
// - Вызывает netpoll для проверки I/O
// - Запускает GC если нужно
```

## Частые вопросы на собеседованиях

**Q: Что такое GMP?**
A: G (горутина) — единица работы. M (machine) — OS поток. P (processor) — логический процессор с локальной очередью. GOMAXPROCS определяет количество P.

**Q: Что такое work stealing?**
A: Когда P исчерпал свою очередь, он крадёт половину горутин из очереди случайного другого P. Обеспечивает балансировку нагрузки.

**Q: Почему локальная очередь — ring buffer на 256?**
A: Компромисс между эффективностью (нет аллокаций, нет мьютекса для доступа своим P) и переполнением (излишек уходит в GRQ).

**Q: Что произойдёт если горутина делает blocking syscall?**
A: P отсоединяется от M и находит (или создаёт) другой M. Горутины на этом P не блокируются.

**Q: Что делает sysmon?**
A: Фоновый мониторинг: вытеснение долгих горутин, отсоединение P от syscall-блокированных M, netpoll, триггер GC.
