# GC: Tri-Color Mark-Sweep (до Go 1.26)

## Обзор

Go использует concurrent, tri-color, mark-sweep GC с write barrier. Основная цель — минимальные паузы (sub-millisecond STW).

## Tri-Color алгоритм

```
Три цвета объектов:
- Белый: не посещён (кандидат на удаление)
- Серый: посещён, но потомки не проверены
- Чёрный: посещён, все потомки проверены

Начало: все объекты белые
Roots (стек, глобальные) → серые

Цикл:
1. Взять серый объект
2. Для каждого указателя в нём: если указывает на белый → покрасить в серый
3. Покрасить текущий объект в чёрный
4. Повторить пока есть серые

Конец: все чёрные — живые, все белые — мусор → освободить
```

```
     Белый         Серый         Чёрный
    ┌─────┐      ┌─────┐      ┌─────┐
    │  E  │      │  B  │──────│  A  │
    │  F  │      │  C  │      └─────┘
    └─────┘      └─────┘
     мусор?      в процессе    живой
```

## Фазы GC

```
1. Mark Setup     [STW ~10-30μs]
   - Включить write barrier
   - Подготовить workers

2. Marking        [Concurrent]
   - GC workers работают параллельно с мутатором
   - Используют 25% CPU (GOGC)
   - Write barrier перехватывает изменения указателей

3. Mark Termination [STW ~10-30μs]
   - Обработать оставшиеся серые объекты
   - Выключить write barrier

4. Sweeping       [Concurrent]
   - Освобождение белых объектов
   - Происходит лениво при аллокации
```

### Write Barrier

Зачем: GC работает параллельно с программой. Программа может изменить указатель ПОСЛЕ того, как GC проверил объект. Write barrier перехватывает это.

```go
// Упрощённо: при записи указателя
// Go использует "hybrid write barrier" (Go 1.8+)
func writeBarrier(dst *unsafe.Pointer, src unsafe.Pointer) {
    shade(*dst) // покрасить старое значение в серый (deletion barrier)
    shade(src)  // покрасить новое значение в серый (insertion barrier)
    *dst = src
}
```

### Триггер GC

```go
// GC запускается когда:
// 1. Heap вырос на GOGC% с прошлого цикла (default GOGC=100 → удвоение)
// 2. Прошло > 2 минуты с последнего GC
// 3. runtime.GC() вызван явно

// GOGC=100: GC после удвоения heap
// GOGC=200: GC после утроения heap (реже, больше памяти)
// GOGC=off: GC выключен (осторожно!)

// GOMEMLIMIT (Go 1.19+): жёсткий лимит памяти
// GOMEMLIMIT=1GiB — GC станет агрессивнее при приближении к лимиту
```

## Частые вопросы на собеседованиях

**Q: Какой GC в Go?**
A: Concurrent, tri-color, mark-sweep с hybrid write barrier. Sub-millisecond STW паузы.

**Q: Что такое write barrier?**
A: Инструментация записи указателей для корректности concurrent marking. Перехватывает изменения указателей пока GC работает.

**Q: Что такое GOGC?**
A: Процент роста heap, после которого запускается GC. GOGC=100 (default) — GC при удвоении. GOGC=off — отключить.

**Q: Что такое GOMEMLIMIT?**
A: Мягкий лимит памяти (Go 1.19+). GC становится агрессивнее при приближении к лимиту, предотвращая OOM.

**Q: Почему Go не использует generational GC?**
A: Write barrier для поколений слишком дорогой из-за частых записей указателей в Go (escape analysis, горутинные стеки). Но Green Tea GC (1.26) наконец добавляет поколения.
