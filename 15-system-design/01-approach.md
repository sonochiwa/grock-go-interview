# Подход к System Design Interview

## Пошаговый план

### Шаг 1: Уточнение требований (2 мин)

**Функцио��альные требования** — ЧТО делает система:
```
- Какие основные use cases?
- Кто пользователи? (B2B, B2C, internal)
- Какие операции: CRUD? Real-time? Batch?
- Нужна ли аутентификация?
- Какие данные на входе/выходе?
```

**Нефункциональные требования** — КАК работает система:
```
- Сколько пользователей? DAU/MAU?
- Какой RPS ожидается? Read/Write ratio?
- Latency requirements? (p50, p99)
- Availability target? (99.9%? 99.99%?)
- Consistency requirements? (strong/eventual)
- Data retention? Сколько хранить?
- Географическое распределение?
```

**Примеры уточняющих вопросов:**
```
"Нужна ли real-time доставка или eventual consistency ок?"
"Какой масштаб: тысячи или миллионы пользователей?"
"Read-heavy или write-heavy нагрузка?"
"Нужна ли мультирегиональность?"
"Какие SLA по доступности?"
```

### Шаг 2: Back-of-Envelope расчёты (3 мин)

```
"Давайте прикинем нагрузку..."
- DAU: 10M users
- Each user: ~10 requests/day
- RPS: 10M * 10 / 86400 ≈ 1200 RPS (avg)
- Peak: 3-5x avg ≈ 5000 RPS
- Storage: 10M users * 1KB = 10GB (metadata)
- Bandwidth: 5000 * 10KB = 50 MB/s
```

### Шаг 3: High-Level Design (5 мин)

Рисуй блок-схему:
```
[Client] → [Load Balancer] → [API Gateway]
                                  ↓
                          [Service Layer]
                         /       |       \
                   [Cache]   [Database]   [Queue]
                              /     \
                         [Primary] [Replica]
```

**Называй компоненты и объясняй ЗАЧЕМ каждый:**
- "LB для распределения нагрузки и failover"
- "Кэш для снижения нагрузки на БД, read-heavy система"
- "Очередь для async обработки, чтобы не блокировать пользователя"

### Шаг 4: Deep Dive (15 мин)

Обычно интервьюер спросит "расскажи подробнее о X". Будь готов к:

- **Data model**: какие таблицы, какие индексы, какие запросы
- **API design**: endpoints, request/response format
- **Конкретный компонент**: как работает кэш, как шардим БД
- **Сложный flow**: что происходит шаг за шагом

### Шаг 5: Масштабирование (5 мин)

```
"При росте до 100M пользов��телей bottleneck будет в..."
- Database: шардинг по user_id
- Cache: распределённый Redis cluster
- Service: гориз��нтальное масштабирование, stateless
- Hot spots: consistent hashing
```

### Шаг 6: Trade-offs и итоги (5 мин)

```
"Мы выбрали X потому что Y, но trade-off в Z"
"С бОльшим временем я бы добавил: мониторинг, rate limiting, geo-distribution"
```

## Что оценивают

| Хорошо | Плохо |
|--------|-------|
| Задаёшь вопросы | Сразу рисуешь архитектуру |
| Обсуждаешь trade-offs | "Это единственный правильный подход" |
| Количественные оценки | "Ну, примерно много" |
| Называешь конкретные технологии | Абстрактные "database", "cache" |
| Признаёшь что не знаешь | Придумываешь на ходу |
| Итерируешь дизайн | "��разу делаем на 1B пользователей" |

## Чего избегать

1. **Не начинай с микросервисов** — начни с монолита, масштабируй по необходимости
2. **Не overengineer** — "нам не нужен Kafka для 100 RPS"
3. **Не молчи** — думай вслух, объясняй каждое решение
4. **Не забывай про failure modes** — "что если этот сервис упадёт?"
5. **Не пропускай монитори��г** — хотя бы упомяни observability
