# I/O

## File Descriptors

```
fd = целое число, указатель на открытый файл/сокет/pipe

Стандартные:
  0 = stdin
  1 = stdout
  2 = stderr

Каждый процесс имеет таблицу fd → file description (в ядре)

Лимиты:
  ulimit -n          # per-process (default 1024)
  cat /proc/sys/fs/file-max   # system-wide

  # Увеличить
  ulimit -n 65535
  # Или в systemd: LimitNOFILE=65535
  # Или в /etc/security/limits.conf:
  # * soft nofile 65535
  # * hard nofile 65535

Проблема "too many open files":
  lsof -p <pid> | wc -l    # сколько fd у процесса
  # Утечка: забыли Close() → fd накапливаются
  # В Go: defer resp.Body.Close() !!!
```

## I/O Models

```
1. Blocking I/O (default):
   read() → блокирует поток до получения данных
   Один поток на соединение → не масштабируется

2. Non-blocking I/O:
   read() → -1 + EAGAIN если нет данных
   Нужен polling (busy wait) → CPU waste

3. I/O Multiplexing:
   select/poll/epoll → мониторить МНОГО fd одним потоком
   "Скажи мне когда на одном из этих fd будут данные"

4. Async I/O (io_uring):
   Submit request → kernel обработает → уведомит о завершении
   Zero-copy, no syscall per I/O

Go runtime: non-blocking fd + epoll (Linux) / kqueue (macOS)
  netpoll: горутины "блокируются" но OS thread свободен
```

## epoll (Linux)

```
epoll = эффективный I/O multiplexer для большого кол-ва fd

API:
  epoll_create()     → создать epoll instance
  epoll_ctl(ADD/DEL) → добавить/удалить fd для мониторинга
  epoll_wait()       → ждать событий на fd (блокирующий)

vs select/poll:
  select: O(n) — проверяет ВСЕ fd каждый раз, лимит 1024 fd
  poll:   O(n) — нет лимита, но всё равно O(n) scan
  epoll:  O(1) — возвращает только ГОТОВЫЕ fd, нет лимита

  10K connections:
    select: scan 10K fd → найти 3 готовых
    epoll:  вернуть только 3 готовых

Edge-triggered vs Level-triggered:
  Level (default): "fd готов" — уведомляет пока есть данные
  Edge: "fd СТАЛ готовым" — уведомляет один раз
  Edge эффективнее, но нужно читать полностью (до EAGAIN)

Go netpoller:
  Каждый сокет → non-blocking + epoll
  Горутина делает Read() → netpoller ставит в epoll → горутина паркуется
  Данные пришли → epoll_wait → горутина просыпается
  → Тысячи горутин на одном OS thread
```

## io_uring (Linux 5.1+)

```
Async I/O через shared ring buffers между user space и kernel

Submission Queue (SQ): user → kernel (запросы)
Ring Buffer (CQ): kernel → user (результаты)

Преимущества:
  - Нет syscall per I/O (shared memory)
  - Batch submit (множество операций за раз)
  - Zero-copy
  - Поддержка: read, write, fsync, connect, accept, send, recv

Пока Go runtime не использует io_uring (epoll),
но есть библиотеки: github.com/iceber/iouring-go
```

## Pipes и Redirect

```bash
# Pipe: stdout → stdin
ls -la | grep ".go" | wc -l
#  ls → pipe → grep → pipe → wc

# Redirect
command > file.txt      # stdout → file (overwrite)
command >> file.txt     # stdout → file (append)
command 2> error.txt    # stderr → file
command 2>&1            # stderr → stdout
command &> all.txt      # stdout + stderr → file

# Process substitution
diff <(sort file1) <(sort file2)

# Here document
cat <<EOF > config.yaml
key: value
port: 8080
EOF

# Named pipe (FIFO)
mkfifo /tmp/mypipe
echo "hello" > /tmp/mypipe &   # producer
cat /tmp/mypipe                 # consumer → "hello"
```

## Частые вопросы

**Q: Почему Go быстрый при 100K+ соединениях?**
A: Go netpoller = epoll + горутины. Каждое соединение = горутина (2-8KB стек), не OS thread (1-8MB стек). epoll мониторит все fd → O(1). Результат: 100K горутин на одном процессе.

**Q: Что такое splice / sendfile?**
A: Zero-copy I/O. sendfile(): disk → kernel buffer → network (без copy в user space). splice(): pipe → socket. Go: io.Copy() использует splice() если возможно.

**Q: Блокирующий vs неблокирующий I/O — что лучше?**
A: Для серверов — non-blocking + event loop (epoll). Go прячет это за blocking API: горутина "блокируется", но OS thread свободен для других горутин. Лучшее из двух миров: простой код + эффективный I/O.
